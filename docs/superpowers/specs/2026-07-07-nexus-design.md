# NEXUS Design — NOTES mode, dashboard pages, quests, outline, find&replace

Date: 2026-07-07 · 5-agent design workflow (notes-mode architect @xhigh, dash-pages, skills-depth, features, judge @xhigh). Ships as v0.8.0.

## User brief (verbatim)
> "take it even further with different dashboard style and even more features… lets add a designated notes tab beside the brain tab but dont stop here keep adding more then just this feature… go the full length NONSTOP… most complex second brain app on the market"

## Mode tab decision (judge)
Tab order: BRAIN, NOTES, DASHBOARD, SKILLS with Ctrl+1/2/3/4 — NOTES takes Ctrl+2 and shifts DASHBOARD→Ctrl+3, SKILLS→Ctrl+4. Rationale: (1) the user explicitly asked for the notes tab 'beside the brain tab', fixing its ordinal position at 2; (2) TopBar.jsx:19 derives every tooltip as Ctrl+(index+1) from the MODES array, so position and shortcut must agree or the derivation breaks — keeping DASHBOARD on Ctrl+2 while NOTES sits second would ship a permanent 2nd-tab≠Ctrl+2 mismatch forever; (3) v0.7.0 is a pre-release single-user app, so the migration cost is one session of muscle memory versus a lifetime of inconsistency. Implementation is exactly: MODES array insert in TopBar.jsx, key array ['brain','notes','dashboard','skills'] at App.jsx:223, the two palette hints at App.jsx:525-526 renumbered plus a new 'Mode: Notes · Ctrl+2' action, and — critically — every mode switch (keyboard, TopBar onMode, palette, fly-to) routed through the new dirty-guarded changeMode() so the docked/overlay reader remount can never eat a draft; F-key focus toggle gains a mode==='brain' guard at App.jsx:228. Canvas pause needs zero changes (paused={mode!=='brain'} already covers the new mode).

## Cuts
- Draggable column widths (NOTES 'could') — fixed 232/324px is fine for a 104-note vault; add only after a real overflow complaint. Its validateNotesUi widths-clamp goes too.
- CHRONICLE tab + level-up 'chronicle' store — QUESTS is the strictly better skills investment (live loop vs retrospective log); a second store, a level-watching App effect, and a timeline UI for ~40 rows is not v0.7.0 material. Defer whole; nearestUnlocks spotlight can ride into a future release.
- clusterMomentum panel (ANALYTICS) — redundant: existing CLUSTERS bars already show composition, tagMomentum already shows the recency signal, and snapshots carry no per-cluster history so 'momentum' is just mtime re-dressed. Cut from insights.mjs scope.
- Palette search operators (folder:/tag:/task: in Ctrl+K) — redundant this release: the new NOTES mode rail IS folder/tag scoping with a filter box using identical matchFilter semantics. Revisit only if palette-scoped search is still missed after living with NOTES mode.
- Export note as styled HTML — only proposal requiring a new main-process IPC + preload surface + save-dialog trust boundary; nice-to-have output channel with zero synergy with the rest of NEXUS. Defer to v0.8.
- Vault time scrubber — self-marked 'could'; snapshot history is weeks old at best ('collecting history' empty state would be the main screenshot), and the existing BRAIN GROWTH panel already plots the same series. Cut.
- Per-proposal path errors corrected, not shipped: 'tests/' directories and lib-adjacent *.test.mjs files (node --test only discovers root test/), and 'src/renderer/styles' (does not exist — all CSS goes in src/renderer/index.css).

## Risks
- Dirty-draft transition matrix is the release's #1 correctness risk: NoteReader REMOUNTS when moving between the docked slot and the overlay slot, silently dropping drafts unless every path routes through the new changeMode()/flyToBrain guards. Must manually test all four: brain→notes with dirty overlay, notes→brain with dirty docked, fly-to from a context menu on a different row, Esc in notes mode.
- OneDrive vs birthtime: sync-engine rehydration can rewrite file creation times, so sort-by-created may mislead on this vault. The ||mtime fallback covers zeroed birthtime but not wrong values — label the sort 'created (fs)' and accept; do not build workarounds.
- Find & Replace is still the highest-risk write path even with rails: the undo snapshot lives in modal component state and dies on close — toast UNDO must be used before dismissing (document in-modal); sequential 104 readNote calls on OneDrive can stall on AV/file locks (the k/104 counter is the mitigation, not a timeout). If any rail gets cut for time, cut the whole feature instead.
- questBonusXp feeds level = sqrt(xp/100), and the unlock-diff toast effect (App.jsx:185) diffs on evaluated — a quest completion can cascade a skill-unlock toast in the same tick; the announced ref handles duplication but test the spread order so questBonusXp survives mergeUsageStats. Also note store.js NAME_RE (lowercase-only) and appdata.js counter NAME_RE (camelCase allowed) are DIFFERENT regexes — 'notes-ui'/'dash-ui'/'quests' vs 'vaultEdit' each pass their own, don't mix them.
- makeExcerpt imported by the main-process indexer from renderer lib is fine (parseTasks precedent at vault-indexer.mjs:5) but puts excerpt logic on the hot scan path for every reindex — keep it regex-cheap, never let it grow markdown parsing.
- DashboardMode.jsx absorbs ~10 new panels across 4 pages in one file on top of 15 existing — acceptable with per-page render functions, but if it crosses ~900 lines mid-build, split pages into modes/dash/*.jsx at that moment rather than refactoring at the end.
- Renumbering Ctrl+2/3/4 invalidates any user muscle memory and the two hardcoded palette hints (App.jsx:525-526) — grep for every 'Ctrl+2'/'Ctrl+3' string literal (tooltips, docs, README) in the same commit as the TopBar change so no stale hint ships.

## NOTES mode proposals

### Mode plumbing: 'notes' as 4th mode, Ctrl+2, tab between BRAIN and DASHBOARD [must, ~35L]

**Value:** Makes the notes workspace a first-class mode reachable by tab and keyboard, with the whole existing mode machinery (canvas pause, palette actions, Esc ladder) working unchanged.

**Design:** TopBar.jsx: MODES becomes [{id:'brain',label:'BRAIN'},{id:'notes',label:'NOTES'},{id:'dashboard',label:'DASHBOARD'},{id:'skills',label:'SKILLS'}] — data-tip already derives Ctrl+N from index, zero other TopBar changes. App.jsx keyboard handler: `if (e.ctrlKey && ['1','2','3','4'].includes(e.key)) { changeMode(['brain','notes','dashboard','skills'][+e.key-1]) }`. DECISION: renumber to Ctrl+2 (notes=2, dashboard=3, skills=4) — justification: v0.7.0 pre-release single-user app, muscle-memory migration cost is one session; a permanent ordinal mismatch (2nd tab = Ctrl+4) ships forever and breaks the index-derived tooltips. NEW `changeMode(m)` in App.jsx replaces bare setMode everywhere (keyboard, TopBar onMode, palette actions): `const changeMode = (m) => { if (m === mode) return; if (!guardDirty()) return; kbRef.current.dirty = false; setMode(m) }` — this closes the draft-loss hole that opens once the reader lives inside a mode's tree (React remounts NoteReader when it moves between the notes-mode slot and the overlay slot, dropping an unsaved draft without this guard; mirrors the existing goBack() dirty-clear pattern at App.jsx:305). Canvas pause needs NOTHING: `paused={mode !== 'brain'}` and the `.stage dimmed` class already key off mode. Focus mode: add `k.mode === 'brain'` to the F-key condition (App.jsx:228) so F can't hide chrome in notes mode. Palette: update the two existing 'Mode:' actions' hints (Ctrl+3/Ctrl+4) and add `{ label: 'Mode: Notes', hint: 'Ctrl+2', run: () => changeMode('notes') }`. Render branch in App.jsx after the brain block: `{mode === 'notes' && <NotesMode ...props (see workspace feature)/>}`. Esc ladder unchanged: overlay → focus → selected → hover; in notes mode Esc with a selected note closes the docked reader to the empty state, which is consistent with the brain-mode contract.

**Files:** src/renderer/App.jsx, src/renderer/components/TopBar.jsx

**Tests:** Manual: Ctrl+1..4 cycle modes; edit a note in notes mode, press Ctrl+1 → confirm dialog appears; cancel keeps draft, confirm discards. TopBar tooltips show renumbered keys automatically.

### lib/notesmode.mjs — pure scope/sort/excerpt/tag-count logic + node --test suite [must, ~210L]

**Value:** All list-column behavior (smart sections, sorting, filtering, snippets, word counting) as tested pure functions, per the house rule: pure logic in lib/*.mjs, UI stays thin.

**Design:** New src/renderer/lib/notesmode.mjs exporting: (1) `scopeNotes(notes, scope, {pins, dailyFolder})` — scope is `{kind:'all'|'recent'|'daily'|'pinned'|'folder'|'tag', value?}`: 'all' → notes; 'recent' → top 30 by mtime desc; 'daily' → folder === dailyFolder; 'pinned' → pins order preserved (map pins→note, filter live); 'folder' → note.folder === value; 'tag' → tags includes value. (2) `filterNotes(notes, q)` — reuses matchFilter from graph.mjs verbatim: `notes.filter(n => matchFilter(n, {q, folders:[], types:[], statuses:[], tags:[]}))` so search semantics are identical to the brain sidebar (title+tags+type haystack). (3) `sortNotes(notes, key, dir)` — key ∈ 'mtime'|'ctime'|'title'|'wordCount'; title uses localeCompare, numerics `(b[key]||0)-(a[key]||0)`, dir flips, ALWAYS returns a copy (never mutate graph.notes). (4) `tagCounts(notes)` → sorted [{tag,count}] desc (topTags in dashboard.mjs caps at 24 and is shaped for the dashboard panel; the rail needs the full list — 6-line reduce, honest duplication). (5) `folderCounts(notes)` → Map folderId→count. (6) `makeExcerpt(content)` — first non-empty line that isn't a heading/hr/frontmatter remnant, strip `[[x|y]]`→y, `[[x]]`→x, `**`/`*`/`` ` ``, slice 140 (consumed by the indexer, tested here). (7) `countWords(s)` — same regex as the indexer (`split(/\s+/).filter(Boolean).length`) for the live statusline. (8) `validateNotesUi(stored, folders)` — sanitize the persisted 'notes-ui' blob: unknown scope kind or dead folder/tag value → {kind:'all'}, unknown sort key → 'mtime'; null-safe for first run. New test/notesmode.test.mjs: every scope kind incl. pinned-order preservation and dead-pin filtering, sort stability + copy semantics, excerpt edge cases (frontmatter-only note, heading-first note, wikilink stripping, empty file → ''), validateNotesUi against garbage input. Keeps 81 existing tests green (no existing lib touched except additive indexer change below).

**Files:** src/renderer/lib/notesmode.mjs, test/notesmode.test.mjs

**Tests:** node --test test/notesmode.test.mjs — full coverage of scoping, sorting, excerpt, and persistence validation with no DOM.

### Indexer: ctime + excerpt fields on every note [must, ~12L]

**Value:** Enables sort-by-created and list-row snippets with ZERO per-row readNote IPC — content is already in memory during scan; without this the list column would storm the disk with 104 reads on every render.

**Design:** vault-indexer.mjs scanVault(): the existing `fs.stat(abs)` at line 58 already fetches the stat object — capture birthtime in the same try block: `const st = await fs.stat(abs); mtime = Math.round(st.mtimeMs); ctime = Math.round(st.birthtimeMs) || mtime` (Windows/NTFS has real birthtime; the `|| mtime` fallback covers filesystems that zero it). Add to the notes.push object (line 73): `ctime: ctime ?? mtime` and `excerpt: makeExcerpt(content)` importing makeExcerpt from ../renderer/lib/notesmode.mjs (main already imports renderer libs — parseTasks precedent). MUST be computed at push time since `_content` is stripped before the graph is sent to the renderer. Extend test/indexer.test.mjs: fixture-vault notes carry numeric ctime and non-empty excerpt; frontmatter-only fixture yields excerpt ''.
